# 前后端总体设计与开发计划

## 1. 文档目标

本设计文档用于统一本项目接下来界面开发、前后端联调和阶段实施路线。

当前项目现状：
- 已完成本地摄像头/网络视频流接入
- 已完成 YOLO 人体/手机检测
- 已完成 MediaPipe 姿态识别与课堂行为规则判断
- 已完成匿名学生跟踪、专注度评分、OpenCV 本地预览
- 后端 HTTP 接口目前仍较少，前端尚未正式搭建

接下来目标：
- 在一台电脑上同时运行前端和后端
- 将当前检测能力服务化，供网页前端实时调用
- 先完成可演示的实时监测系统，再逐步补齐记录、分析、设置和 AI 功能

---

## 2. 总体产品形态

本项目采用：
- 本地前后端分离
- 单机部署运行
- 后端负责检测、统计、截图记录、AI 数据组织
- 前端负责界面展示、控制操作、图表分析和交互

不采用：
- 微服务
- API 网关
- 云端部署优先路线
- WebRTC 重型音视频方案

推荐运行形态：
- 后端：`FastAPI`
- 前端：`Vue 3 + Vite`
- 视频显示：`MJPEG` 流
- 实时数据：`WebSocket`
- 普通数据：`REST API`

---

## 3. 总体架构设计

### 3.1 架构分层

#### 检测内核层
- 摄像头读取
- 检测线程
- 姿态分析
- 跟踪
- 行为识别
- 专注度评分

#### 后端服务层
- 检测会话管理
- 当前课堂状态缓存
- 实时数据推送
- 视频流输出
- 异常截图记录
- 历史记录接口
- 分析接口
- AI 辅助接口

#### 前端展示层
- 实时检测页
- 检测记录页
- 分析页面
- 设置页
- AI 助手

### 3.2 数据流

整体数据流如下：

1. 视频源输入到检测内核
2. 检测内核输出当前帧结果
3. 后端将结果写入“当前课堂状态”
4. 后端同时：
- 推送实时状态到 WebSocket
- 输出视频流到 MJPEG 接口
- 记录异常截图和课堂事件
5. 前端读取：
- 视频流
- 当前状态
- 记录数据
- 分析数据
6. AI 功能基于结构化状态和历史统计生成回答或总结

## 4. 页面总体设计

前端页面统一采用一套信息架构，不再分别设计两套导航体系。

### 4.1 一级页面

一级页面固定为：
- 实时检测
- 检测记录
- 分析统计
- 设置
- AI 助手

### 4.2 全局导航规范

顶部导航负责一级页面切换：
- 实时检测
- 检测记录
- 分析统计
- 设置
- AI 助手

左侧导航只在需要时展示当前页面下的二级功能，不再重复一级页面。

## 6. 前端技术设计

### 推荐技术栈
- `Vue 3`
- `Vite`
- `Vue Router`
- `Pinia`
- `Axios`
- `ECharts`

### 推荐目录结构

```text
frontend/
  src/
    api/
    assets/
    components/
    layouts/
    pages/
      realtime/
      records/
      analytics/
      settings/
    router/
    stores/
```

## 7. 后端服务化设计

当前检测逻辑仍主要运行在本地预览入口中，接下来要做的是“服务化接入前端”。

### 7.1 后端新增能力

需要新增：
- 检测会话管理
- 当前状态缓存
- 视频流接口
- WebSocket 实时推送
- 异常截图保存
- 记录查询接口
- 分析接口
- 设置接口
- AI 对话接口

### 7.2 后端内部模块建议

建议新增以下服务层模块：

```text
backend/app/services/
  session/
    session_manager.py
  runtime/
    classroom_state.py
  stream/
    mjpeg_stream.py
  records/
    alert_store.py
  analytics/
    analytics_service.py
  ai/
    ai_assistant_service.py
```

### 7.3 核心运行对象

建议增加全局运行态对象：
- 是否运行
- 当前视频源
- 当前检测结果
- 当前课堂统计
- 最新异常列表
- 当前检测时长
- 最新推理 FPS

前端所有实时页能力都建立在这个对象之上。

## 8. 接口分层设计

不需要 API 网关，但必须做清晰的接口分层。

### 8.1 控制接口

- `POST /api/v1/session/start`
- `POST /api/v1/session/stop`
- `GET /api/v1/session/status`

### 8.2 实时数据接口

- `GET /api/v1/classroom/current`
- `WS /ws/classroom/realtime`
- `GET /api/v1/video/stream`

### 8.3 记录接口

- `GET /api/v1/records`
- `GET /api/v1/records/{id}`
- `GET /api/v1/alerts/current`

### 8.4 分析接口

- `GET /api/v1/analytics/attention-trend`
- `GET /api/v1/analytics/behavior-summary`
- `GET /api/v1/analytics/activity-trend`

### 8.5 设置接口

- `GET /api/v1/settings`
- `PUT /api/v1/settings`

### 8.6 AI 接口

- `POST /api/v1/ai/chat`
- `GET /api/v1/ai/summary/current`

## 13. 项目实施建议

### 建议一
- 先做“能演示”的最小闭环
- 不要同时铺开所有页面细节

### 建议二
- 接口字段先冻结，再并行开发
- 前端和后端都按固定字段走

### 建议三
- 第一阶段不优先数据库
- 可先用内存状态 + 文件记录

### 建议四
- 第一阶段不优先邮箱
- 实时检测与记录展示比邮件更重要

---

## 14. 结论

本项目接下来应采用：
- 单机前后端分离
- 后端检测内核服务化
- 前端围绕实时检测页优先开发
- 先打通视频流、实时状态和异常记录
- 再扩展分析统计、设置和 AI 助手

总体实施顺序应为：

1. 实时检测页
2. 检测记录页
3. 分析统计页
4. 设置页
5. AI 助手

这条路线最符合当前项目状态，也最适合中后期演示和答辩。
